# Woolworths Cart Builder

Build the user's cart with the Woolworths MCP tools. Prepare the cart only: never place the order, submit checkout, choose a delivery or pickup window, enter payment details, or confirm a purchase.

## Inputs

Accept grocery items from:

- the `Combined Ingredients` output of the `weekly-meal-prep` skill
- a pasted grocery list
- a JSON ingredient list with `ingredient`, `quantity`, `notes`, and optional Woolworths product links

If more than one plausible grocery list is present, ask which list to use. Otherwise, treat a request to build the cart as authorization to add the supplied items; do not ask for a second blanket confirmation.

If the list comes from a meal plan generated by `weekly-meal-prep` in the current conversation, cart building means the user accepted that plan. Before changing the cart, append the plan to the active private `~/.meal-prep-agent/history.json` if it has not already been saved.

## Required Tools

Use the tools from the `woolworths` MCP server directly. Do not use a browser-control sub-agent or a separate item-adder sub-agent.

Required tools:

- `woolworths_open_browser`
- `woolworths_get_cookies`
- `woolworths_search_products`
- `woolworths_get_product_details` when search data is insufficient
- `woolworths_add_to_cart`
- `woolworths_get_cart`
- `woolworths_navigate`
- `woolworths_close_browser` only when abandoning or explicitly asked to close it

Do not use `woolworths_set_fulfilment`, `woolworths_remove_from_cart`, or `woolworths_update_cart_quantity` unless the user explicitly requests that specific change. Never call fulfilment or cart-removal tools merely to make the meal-prep list match a cart that already contains other products.

If the Woolworths MCP tools are unavailable, stop before changing anything and tell the user that the `woolworths` MCP server must be enabled. Do not silently fall back to browser automation.

## Workflow

1. Normalize the grocery list into one record per item: ingredient, requested amount, notes, preferred product URL or stockcode, and dietary constraints.
2. Call `woolworths_open_browser` with `{"request":{"headless":false}}` so the user retains a visible Woolworths session.
3. Ask the user to sign in and select their store or location in that browser when needed. Never request, read, or enter credentials.
4. After the user has completed session setup, call `woolworths_get_cookies`. Do not call catalogue or trolley tools until cookie capture succeeds.
5. Call `woolworths_get_cart` before adding anything. Record the existing item count and totals; preserve all existing items.
6. Resolve and add each grocery item using the product-selection rules below. Process pantry-check items only when the user has indicated they need them; otherwise list them under `Needs Review` without adding them.
7. Call `woolworths_get_cart` after all additions. Compare the result with the initial cart and the per-item mutation results. Retry a failed read once; never retry an add blindly because that can duplicate an item.
8. Call `woolworths_navigate` with `{"request":{"url":"https://www.woolworths.com.au/shop/mylist"}}` to leave the visible browser at the trolley review surface. If navigation fails, keep the browser open and tell the user to open the cart icon manually.
9. Present the verified summary and hand control to the user for review and purchase. Leave the browser open unless the user asks to close it.

## Product Selection

For each item:

1. If a credible Woolworths product link or stockcode is supplied, extract the stockcode and call `woolworths_get_product_details` first.
2. Otherwise call `woolworths_search_products` with a concise, specific term and a small first page, normally 5-10 results sorted by `TraderRelevance`.
3. Exclude products whose `is_available` value is `false`.
4. Compare name, package size, unit, price, dietary requirements, and quantity already in the trolley. Use `woolworths_get_product_details` if the search result lacks information needed for a safe choice.
5. Choose the closest practical product and calculate the number of packs needed to meet the requested amount. Prefer the smallest reasonable overage; avoid excessive bulk. For count-based or variable-weight produce, use a conservative practical quantity and flag uncertainty.
6. Call `woolworths_add_to_cart` once with the selected stockcode and required pack count. The input quantity is a pack count from 1 to 99, not grams or millilitres.
7. Record the tool's success, returned cart summary, chosen product, stockcode, pack count, estimated supplied amount, price, and any mismatch.

Do not infer a pack count when package information is ambiguous. Put the item in `Needs Review` instead.

## Substitution Policy

Add obvious, low-impact equivalents without pausing, including:

- Woolworths brand versus a comparable brand
- loose produce versus equivalent packaged produce
- the closest slightly larger pack when an exact quantity is unavailable

Ask before adding a choice that changes:

- protein type
- dietary or allergen characteristics
- fresh versus frozen form
- flavour profile
- a requested brand where the brand is material
- price or supplied quantity materially beyond the request

When user input is required, continue resolving other unambiguous items first, then ask one consolidated question for the unresolved choices.

## Mutation and Verification Rules

- Treat cart calls as real external mutations.
- Never remove or reduce pre-existing cart items.
- Never add an unavailable or low-confidence match just to complete the list.
- Do not repeat `woolworths_add_to_cart` after a timeout or ambiguous response. Read the cart or search the product again and inspect `quantity_in_trolley` before deciding whether another mutation is safe.
- If the product was already present, account for `quantity_in_trolley` and add only what the grocery list still requires. Report it as `already_in_cart` when no addition is needed.
- Consider an item verified only when the add response succeeds and the final cart read is consistent with the mutation. Mark inconsistencies as `Needs Review`.
- Never call checkout, payment, order-confirmation, or fulfilment mutations as part of this skill.

## Final Response

Present only the useful cart handoff:

1. `Cart Added` — product, pack count, and supplied amount
2. `Substitutions` — requested item and chosen equivalent
3. `Needs Review` — pantry checks, ambiguity, or verification mismatch
4. `Unavailable or Failed`
5. `Cart Summary` — final item count and total when returned by the MCP
6. `Next Step` — tell the user the Woolworths browser is open for their review and final purchase

Do not claim success from search results alone. Distinguish verified additions from uncertain mutations and omit empty sections.
